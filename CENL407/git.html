Git Basics

Dr. Tuğberk Kocatekin

Version control systems are tools used to track changes to source code or other collection of files and folders/directories.

On top of maintaining a history of changes, they also facilitate collaboration. Almost all companies are using version control and the standard application they use is git.

Even if you are not in a group setting and working by yourself, version control is very useful. For example, we all used to save every change as a different file while we are writing our drafts for thesis, etc. By using git, we can change only one file and later come back to every other version and see the change we applied onto that file.

When you are collaborating, it is impossible to work on the same exact file. When you try to change a file someone else is changing, the file will be locked and you are not going to be able to do some changes.

Git solves these problems. You can create branches and do development and later you can merge these together into a single repo. There are GUI for Git, however it is best to use the command line. Git is built by Linus Torvalds who also is the creator of Linux kernel.

Snapshots

Git stores these changes in a series of snapshots. In Git terminology, a file is called a blob. A directory is called a tree. The tree maps names to blobs and other trees.

A snapshot is the top-level tree that is being tracked.

How does it work?

Git does not use a linear history. In Git, history is a DAG (directed acyclic graph) of snapshots. This means that each snapshot refers to a set of parents, the snapshots that comes before it. It is usually a set of parents instead of a single parent. The reason is, a snapshot can descend from multiple parents. An example would be a merging of two branches of development.

Git calls there snapshots, commits. Commits are immutable. That means any edit to a commit history creates a new commit and references to the previous one. So every change is tracked.

By using DAG, in Git we can actually change the base version. When we do it linearly, we can only use the previous version as base. However, now we can just start to change stuff by basing changes off of another version.

Git is immutable. You can add stuff but you cannot change any snapshot. Every change is a new snapshot and added to the line.

Branching and merging

Let’s say you are trying to fix a bug and at the same time trying to implement some new feature. We don’t have to do this in the same line of development. We can use branching, and we can work on them in parallel. Therefore we are not going to break the code and if two functionalities are not breaking each other, we can develop them separately.

Here, a branch can be for bug-fix another branch can be for development. Later, we can create another snapshot and merge these two branches into the same line.

How to use it?

We start by writing git init. This will initialize git on the directory which we are in.

Later, we can add files into the staging area by using git add .. This will add everything in that directory to the staging area. We can check what is added by using git status.

Let’s create a new file, add it to the staging area and check what happened by using git status.

git init
touch hello.txt
git add .
git status

Here, we first initialized git in that directory. Later, we created a hello.txtand added everything to git. Lastly, we are running status to see what happened. We can also write git add hello.txt instead of adding everything.

When we initialize git, git actually creates a new _hidden_ directory. We can check this directory by `ls .git` and see what it holds.
On branch master

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   hello.txt

As you can see, we can see that there is a hello.txt file in the staging area. Now, if we want to add this into our git repo, we need to commit. This is done by using a very simple command: git commit.

However, we need to give some information as to what this commit has. Maybe we added a new function to our code, or maybe we fixed a bug. That is why, we can use messages in our commits. By using -m we can add message to it. Let’s commit what we have done above by using git commit -m "First commit".

At this point, if you did not configure your git by using an email and username, it will ask you to do so. Just do them by following the instructions given to you by the error message.

After doing so, run the commit again. This will be an example output.

[master (root-commit) c359dda] First commit
 1 file changed, 0 insertions(+), 0 deletions(-)
 create mode 100644 hello.txt

Let’s give an additional information at this point. Git also stores some metadata which gives us information as to who made which commit, when, message, etc. This is very important especially in group/team settings. Let’s say that a part of a code is changed, by using the logs and metadata of the Git, you can now know who changed it when (timestamp) and why (messages). That is why providing a good commit message is very important.

Let’s now change something in hello.txt and run git status. This will be the output:

On branch master
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        modified:   hello.txt

no changes added to commit (use "git add" and/or "git commit -a")

As you can see here, it will give you the information that hello.txt is modified. That change or modification is not saved yet, that is what the last line indicates. In order to commit, we need to add that. Let’s add it by using git add hello.txt and check the status again.

On branch master
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        modified:   hello.txt

Here, as you can see it tells us these are changes to be committed. Before, it told us that some changes were not staged for commit. By adding them into the staging area we did exactly that. Now, we can commit this change by using the same command we used earlier. When we run git commit, the output is:

[master ab40ee8] new line added
 1 file changed, 1 insertion(+)

Previously, when we first created the file, it was 0 insertion and 0 deletion. Here, since we added some lines into the file, we inserted stuff. If we delete, we will see deletion.

Problems with merging

Sometimes, two files you want to merge can conflict with each other. Maybe one is changing some part of the code which the other code uses. This is called merge conflict.

Git will show you that there is a conflict and lets you decide what to do.

Sha-1 Hash and References

We identify objects by using a hash value. Every snapshot will have a different id which is the hash value. It is a hexadecimal string. However, it is hard to remember. That is why git also stores references. Human readable names are mapped to hashes. These are references. We can refer to snapshots by names.

We know that the snapshots/commits in Git are not mutable, but the references are. That means we can change a reference and it can now refer to some other snapshot.

When you look at the previous examples, you can see the hash value. For example in the last example, it writes [master ab40ee8] where the ab40ee8 is the hash value for that commit.

Committing

We can use git add . or git commit -a to add & commit every changed file. However there may be times that we changed multiple files and want to commit them differently. That is why adding and committing manually may be a better practice in some cases.

Carry on

We can use git log to see everything we have done previously. For example when we run git log, this is the output:

commit ab40ee818618d9dcd9a55871fc38e91f156527cf (HEAD -> master)
Author: Tugberk <tugberk@gmail.com>
Date:   Wed Nov 9 16:47:14 2022 +0300

    new line added

commit c359dda047efcdfe971626f5400d18c2570da89f
Author: Tugberk <tugberk@gmail.com>
Date:   Wed Nov 9 16:34:23 2022 +0300

    First commit
    ```

As you can see, we can see the Author and timestamp. Also we can see the hash of the commit, and the commit messages. We can also get detailed information by using `git cat-file -p hash`. For example here, lets do `git cat-file -p c359dda047efcdfe971626f5400d18c2570da89f`. It will give us more information on that specific commit. You can use the same command and change the hash, eventually you will get the content of the blob file.


When the line gets a little complicated, we might want to add some visual to it. For that, we can use `git log --all --graph --decorate`. However, if you don't have any branches, it will not be that different. When you look at the output, you will see something similar to the line. In there, you will also see that `HEAD -> master`. That means that we are at the _master_ branch. We are there by default. 

## Git checkout

By using this command, we can detach the _HEAD_ and check out certain commits by using the _hash_. Since the hash is too long, we can just use some characters from the top and checkout that commit. `git checkout c359dda`. This is the output:

```bash
Note: switching to 'c359dda'.

You are in 'detached HEAD' state. You can look around, make experimental
changes and commit them, and you can discard any commits you make in this
state without impacting any branches by switching back to a branch.

If you want to create a new branch to retain commits you create, you may
do so (now or later) by using -c with the switch command. Example:

  git switch -c <new-branch-name>

Or undo this operation with:

  git switch -

Turn off this advice by setting config variable advice.detachedHead to false

HEAD is now at c359dda First commit

What happens is that, we have now gone back to that state. If we now go to the directory and try to see the contents of hello.txt, it is now changed. (Actually, it is not changed. Because we went to that commit, we can see the previous version of the file.)

This is very useful. This way, if we have committed some errors, we can go back to that specific file and create snapshots out of this file! Now, if we do git log again; we will see a totally different thing. If we want to see some details, run the long git command we have previously ran; git log --all --graph --decorate. Now, we can see every commit but we will see that our HEAD is changed.

If we want to get to the master (original) we can use git checkout hash to get to the original file and timeline. For simplicity, we can also write git checkout master. After running that, check the file again to see whether it returned to the original state.

If you changed a file and want to change your HEAD pointer without committing them, git will give you an error. You can try that by changing the file and trying to checkout to a different branch to see the error. You can change it forcefully though by using -f.

git diff

This command shows you what changed since the last snapshot. You can also modify the command to show the difference between a specific commit. Let’s say you changed your hello.txt and without doing anything else ran the command git diff hello.txt. You will see what you’ve changed. If you want to see the difference compared to another commit, you can write git diff hash hello.txt hash being the starting digits of a hash you want to compare it to. (you can use HEAD, master instead of hash here again.)

Branching and merging

We can see our branches by using git branch -vv.

Instead of text, lets write some code and see the differences there. Let’s build a very simple file and save it as animal.py.

def animal():
        print('Hello')

def main():
        default()

if __name__ == '__main__':
        main()

Let’s add it to the git repo. We will use git branch to create a branch. When you run this command, it will only output master because it is the single branch we have. We can use -vv to have extra information. If we use git branch cat we will have new branch. When we run that command and run the log command, we will see something different. HEAD will now point to master, and we will see cat near it. Because cat is also referring to the same commit at that point. When we go into the cat branch by using git checkout cat and use the long log command; we will see that HEAD is now pointing to cat instead of master.

At this point, we know how to create a branch. We need to be careful at which point we want to create branches. For example, if you want to create another branch, you should first checkout to master and create the branch from there. So keep in mind where you are at all times. Remember that HEAD will point to your present location.

Let’s say we added another function in the code.

def animal():
    print('Hello')

def cat():
    print('Meow')

def main():
    default()
    cat()

if __name__ == '__main__':
    main()

Let’s say we did some changes in the cat branch. We can later go back to master branch and create a new branch called dog. We can create a branch and checkout to it. However, there is a faster way. We can write git checkout -b dog and create this branch and checkout to it just by this command. Let’s add another function to animal.py, this time for the dog.

def animal():
    print('Hello')

def dog():
    print('Woof')

def main():
    default()
    dog()

if __name__ == '__main__':
    main()

Now we commit this data too. At this point, lets run the long log command and see what happened. Here, we see a structure which has branches in it. You can also run the command like this to get a more clear output: git log --all --graph --decorate --oneline.

Merging

Now, we can merge these two branches to go back to the master line and we will see that changes are written in the same file. For this, we go back to the master and use git merge command. We can either merge both of them to master at the same time by using git merge cat dog or can do them one by one.

In case of a conflict, git will say there is a merge conflict and you need to solve it before committing again.

Additional commands

Most of the time we will not use git only in our computers. For that, we are going to have a remote git repository. So, you will need to get data from that remote or push some data to that remote to add.

For that, first you need to add the repository to your remote. We do this by using git remote add origin URL. After that, when you apply some changes, you will use git push and when you want to fetch changes done to the repository (by others) you will use git fetch. When you fetch you will want to merge it to your own repo. So, at that point instead of using two commands, you can use git pull.

For example if we have one remote, we can use git push origin master:master. This means that I am going to push my master branch to the master branch on the remote repo.

In addition, if you want to download the repository from a remote location to your computer and start working on it, you will use git clone command.
